## Installation

<Tabs>
  <TabItem label="VS Code">
    <Steps>
      1. Install **DevSquad Copilot** by clicking the button below. It will open VS Code and install the plugin automatically.

         <LinkButton href="vscode://chat-plugin/install?source=microsoft/devsquad-copilot" variant="primary" icon="rocket" iconPlacement="start">Install in VS Code</LinkButton>

         <Aside type="note">
           Make sure you are signed in with your GitHub account via the Copilot extension before using DevSquad.
         </Aside>

      2. Use `devsquad` as the default entry point in the chat panel. Use `devsquad.<agent>` for direct invocation.
    </Steps>
  </TabItem>
  <TabItem label="GitHub Copilot CLI">
    <Steps>
      1. Install the plugin:

         ```bash
         copilot plugin install microsoft/devsquad-copilot
         ```

         <Aside type="note">
           Make sure you are authenticated before using DevSquad. Run `gh auth login` if you have not already.
         </Aside>

      2. Use `devsquad` as the entry point via the `--agent` flag:

         ```bash
         copilot --agent devsquad:devsquad
         ```

         For direct agent invocation, use `devsquad:<agent>` (e.g., `devsquad:devsquad.implement`).

         <Aside type="tip" title="Unattended Mode">
           The `--yolo` flag enables unattended mode, where the agent operates autonomously by automatically approving terminal commands, file changes, and tool usage without requiring user confirmation. Use only when you accept autonomous file edits, terminal commands, and tool calls without individual confirmation.

           ```bash
           copilot --agent devsquad:devsquad --yolo
           ```
         </Aside>
    </Steps>
  </TabItem>
</Tabs>

### Verify Installation

After installation, open VS Code and type `@devsquad` in Copilot Chat. You should see the conductor agent respond with phase detection. If using the CLI, run `copilot --agent devsquad:devsquad` and confirm the agent responds. If you see an error, check the [Troubleshooting](/devsquad-copilot/troubleshooting/) page.

## Project Initialization

Run `@devsquad.init` to set up SDD Framework files in your repository:

<Steps>
1. **Config files** — `copilot-instructions.md`, instruction files for specs/ADRs/tasks, coding guidelines, markdownlint config.
2. **Documentation templates** — Templates for features, migrations, envisioning, and ADRs.
3. **Community files** (optional) — `SECURITY.md`, `CONTRIBUTING.md`, `LICENSE`, `CODE_OF_CONDUCT.md`.
</Steps>

After initialization, your project structure will look like this:

<FileTree>
- .github/
  - instructions/
    - specs.instructions.md
    - adrs.instructions.md
    - tasks.instructions.md
    - envisioning.instructions.md
    - migration-specs.instructions.md
    - migration-tasks.instructions.md
    - documentation-style.instructions.md
  - copilot-instructions.md
- docs/
  - envisioning/ (vision artifacts)
  - features/
    - feature-name/
      - spec.md (specification)
      - plan.md (technical design)
      - tasks.md (task decomposition)
  - architecture/
    - decisions/ (ADRs)
  - migrations/ (migration specs)
- .markdownlint.yaml
</FileTree>

## Your First Workflow

<Steps>
1. **Envision** — `@devsquad.envision`
   Capture the strategic vision through structured questions about customer, pain points, and goals.

2. **Kickoff** — `@devsquad.kickoff`
   Structure the project hierarchy with epics and features. Sync with GitHub Issues or Azure DevOps.

3. **Specify** — `@devsquad.specify`
   Write feature specifications with user stories, acceptance criteria, and compliance requirements.

4. **Plan** — `@devsquad.plan`
   Generate technical design artifacts. Create Architecture Decision Records for significant choices.

5. **Decompose** — `@devsquad.decompose`
   Break specifications into user stories and tasks. Create work items on the board.

6. **Implement** — `@devsquad.implement`
   Execute implementation from a task, issue, or work item. Quality gates and branch management included.

7. **Review** — `@devsquad.review`
   Validate the implementation against specs, ADRs, and the plan with independent context.
</Steps>

## Usage Scenarios

The framework supports multiple entry points depending on your workflow — feature-first, architecture-first, board-first, or iterative. See [Usage Scenarios](/devsquad-copilot/framework/#usage-scenarios) for patterns like greenfield projects, feature additions, and migrations.

## Guided vs Direct Mode

<Tabs>
  <TabItem label="Guided">
    Invoke `@devsquad` without specifying an agent. The conductor walks through each phase in sequence, asking Socratic questions and delegating to the right specialist.

    ```
    @devsquad
    ```
  </TabItem>
  <TabItem label="Direct">
    Invoke a specific agent for targeted work. Both modes use the same agents, skills, and quality gates.

    ```
    @devsquad.implement    # Go directly to implementation
    @devsquad.specify      # Go directly to specification
    @devsquad.security     # Run a security assessment
    ```
  </TabItem>
</Tabs>

<Aside type="tip">
  Both modes have access to the same agents, skills, and quality gates. Guided mode adds orchestration and phase transitions. Direct mode gives immediate access to a specific capability.
</Aside>
